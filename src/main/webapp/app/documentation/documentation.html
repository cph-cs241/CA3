Fetching data from google places API

Since we are not able to make google API requests directly from the client we have decided to make the request from the client to our server, from over server to google, then send the result back to the client. It is not the most optimal way of doing it, because we load more traffic onto our server and there is even more steps the information have to travel through to get to the end user, which also makes the application seeming slow.

Unit Test Testing - Facade


We are using an in memory Derby mock database for the JUnit tests. There are only two facade test so far, the reasoning for this is simple, we fetch data directly from google map API. The information is not stored in the DB in the current version of the app, the resulting JSoN is formatted and displayed directly on the client side using AngularJS. So only two simple test were written to check wheter the facade methods are working as they should. The createShopTest adds some users and asserts if they are persisted.

REST-ASSURED TEST


We have implemented REST_Assured framework to test the rest services in our project. It was already setup from the seed wherein it initially tests the authentication for the users. We have added some test cases wherein it will validate the response that we get once we have created a new shop and get all the shops in a given path from the use of ShopRest. From there, we are expecting a status code of 200. We also added some tests for the error handling by checking the status code whenever an invalid path was given wherein we get an error code of 404 which means the requested resource was not found.

Manual Testing with Postman


To manually test our api we are using postman with a given path. For example, http://localhost:8080/seedMaven/api/shop/all which returns all the shops as json objects and http://localhost:8080/seedMaven/api/shop which could take a new shop from json that corresponds to the Shop entity class and persist into the database.
